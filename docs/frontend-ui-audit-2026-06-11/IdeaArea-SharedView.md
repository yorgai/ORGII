## Context

`SharedView` is one of three views inside the IdeaArea page (Trending / Shared / My Ideas). It renders a search field + a vertical list of `<IdeaCard>` items. The page-level layout (`SplitViewLayout`, `DetailPanelContainer`, `InternalHeader`, `ScrollFadeContainer`, `TabPill`, `Placeholder`, `MenuPanel`) already routes through the design system; the audit surface is concentrated in two files: the inline search box in `SharedView` and the entire `IdeaCard` body.

- **1 D2 fix candidate** (focus-ring formula) pending sweep before fix-mechanism decision.
- **0 D3 in-file fixes** — all ~10 `text-[Npx]` hits defer to the existing repo-wide D3 sweep.
- **2 D4 fixes** (search input `aria-label`, button `type="button"`) — small, low-risk, can land together.
- **3 keep-with-reason** decisions recorded (outer card `<button>`, inline filter `<input>`, inner upvote pill `<div>`).
- **1 D5 abstract candidate** (Tag absorption of categorical badges) + 3 watch-list patterns.

**Total recommended changes: 0 source edits in this report.** Landing belongs to:

1. A new "IdeaCard → DS Tag" PR (D1 + D2 + cascading D3 in `IdeaCard.tsx`).
2. The existing D3 typography-scale-sweep PR (covers all remaining `text-[Npx]` here).
3. A small a11y patch PR (2 lines: `aria-label` + `type="button"`).

The user's intuition ("感觉很多原生 html 组件") is half-right: the dominant debt in this page isn't actually raw HTML (most raw elements are correctly kept — DS doesn't cover their shapes). The real debt is the **categorical color lookup tables + raw Tailwind palette colors that re-implement what DS `Tag` already provides**, plus the in-file `text-[Npx]` literals that the repo-wide sweep is already tracking.

## Risks for the landing PR(s)

- **R1 — Tag color set may not cover the 6 categories.** Before substitution, confirm `Tag.color` accepts/maps to the 6 brand categories (productivity / dev-tools / ai / collaboration / utilities / creative). If not, either extend `Tag` (per the skill's "abstract once" rule) or fold the lookup table into `Tag` itself via a sibling `IdeaTag` wrapper.
- **R2 — `bg-blue-500/10` etc. may render slightly differently from the equivalent ORGII semantic token.** Visual parity check required before the PR ships. Side-by-side screenshots of the IdeaArea page recommended.
- **R3 — IdeaCard is also rendered in TrendingView and MyIdeasView.** Any fix to IdeaCard touches all three views. Audit the other two views before landing.
- **R4 — `type="button"` is a non-visual change but flips the default form-submission behavior**, which is the intent. Verify no caller currently depends on submit semantics.
